## 개요 (Abstract)

- 이 CIP는 Coin Metrics의 기존 **Weight 1.0 SV**(CIP-0046)를 유지하면서, **에스크로 기반 최대 획득 가능 Weight +6.5**를 Talos에 추가한다.
- 증분 Weight는 **Accountability Subcommittee의 채택 마일스톤 검증** 후에만 발행(mint)된다.
- 마일스톤 범위:
  - **Canton Coin(CC) 트레이딩/커스터디** 준비 (Talos 플랫폼 경유)
  - **pX 결제**의 Canton 통합
  - 정의된 거래량 및 결제량을 통한 **실제 프로덕션 사용 측정**

---

## 동기 (Motivation)

이 제안은 증분 SV 보상을 **실제 사용과 서비스 품질** — 준비 완료, 실시간 흐름, 운영 성과 — 에 연동하며, 커뮤니티 선례와 일관성을 유지한다.

보상은 두 가지 핵심 이니셔티브로 나뉜다:

### 이니셔티브 1: CC 트레이딩/커스터디 지원

Talos 플랫폼 전반에서 CC를 위한 포괄적 지원을 구축한다:
- **Security Master** 및 어댑터에 CC를 통합하여 주문 라이프사이클, RFQ 흐름, 입출금, 주소 화이트리스트가 기본적으로 작동하도록 함
- Talos 연결 거래소 또는 커스터디안이 CC를 활성화하면, **영업일 5일 이내에 활성화**를 약속하여 고객이 기존 워크플로우를 변경 없이 사용 가능

### 이니셔티브 2: pX 결제

pX는 Talos의 **독립적 에이전시 트레이딩 데스크**로, 기관 고객의 암호화폐 생태계 온보딩을 가속화하는 역할을 한다:
- 상위: 거래소/마켓메이커를 대면
- 하위: 기관 고객을 대면
- 거래소 및 커스터디안 간 신용 및 포스트 트레이드 워크플로우를 관리
- pX는 **오프체인에서 의무를 산정**하고 **양자 결제를 Canton을 통해 라우팅**하여, 카운터파티와의 프라이버시 보호·확인된 결제를 제공
- 이를 통해 **실제 기관 결제 볼륨**을 Canton에 유입하고, 네트워크의 프라이버시 및 감사 가능성을 시연

## 마일스톤 및 Weight 획득 조건 (Deliverables)

총 7개의 마일스톤으로 구성되며, **CC 트레이딩** 트랙(4건)과 **pX 결제** 트랙(3건) 두 갈래로 나뉜다.

### CC 트레이딩 트랙 (최대 +3.5)

#### D1. CC 지원 및 준비 완료 (+1)

| 항목 | 내용 |
|------|------|
| **달성 조건** | **플랫폼 CC 준비 완료**: CC 시장 데이터 캡처 설정, 메트릭 산출 구성 / CC를 Security Master에 추가, 라우팅·리스크 파라미터 구성 / 거래소 및 OTC 어댑터 CC 대응 (주문 라이프사이클 & RFQ 스키마) / 커스터디 모듈 CC 대응 (주소·화이트리스트, 입출금 콜백) |
| **UAT/샌드박스 검증** | UAT 데모: 거래소 거래 1건, OTC RFQ 1건, 커스터디안 이전 1건 / 고객 샌드박스 환경에서 기본 CC 지원 추가 |
| **기한** | CIP 승인 후 **90일** |
| **획득 Weight** | **+1** |

#### D2. CC 지원 가속 보너스 (최대 +1)

| 항목 | 내용 |
|------|------|
| **달성 조건** | D1을 빨리 완료할수록 보너스 Weight를 더 받음 |
| **기한** | CIP 승인 후 **90일** |
| **획득 Weight** | 45일 이내 완료 시 **+1** (만점), 90일에 완료 시 **+0**, 선형 비례 (예: 54일 → +0.8) |

#### D3. CC 거래소 활성화 (+0.5)

| 항목 | 내용 |
|------|------|
| **달성 조건** | Talos 연결 거래소에 CC가 상장되면, **영업일 5일 이내에 Talos에서 CC 활성화** / CC 지원 발표 보도자료 또는 마케팅 페이지 게시 / 파트너가 CC를 상장하면 5일 내 활성화 후 프로덕션 거래 1건 기록 |
| **기한** | Talos 연결 거래소가 CC를 추가한 시점부터 **영업일 5일 이내** |
| **획득 Weight** | **+0.5** |

#### D4. CC 채택 보너스 (최대 +1)

| 항목 | 내용 |
|------|------|
| **달성 조건** | Talos를 통해 CC가 거래된 **누적 금액** 기준으로 보너스 부여 (거래소 및 OTC 합산) |
| **증빙 방식** | Talos CC 거래 기록 (타임스탬프 포함 집계 및 산출 방법론)을 TWG에 제출하여 검증 |
| **기한** | 첫 거래소 Go Live 후 **180일** |
| **획득 Weight** | **$2,500만(25M) CC 거래량당 +0.5**, 최대 **+1** |

### pX 결제 트랙 (최대 +3)

#### D5. pX 결제 지원 및 준비 완료 (+1)

| 항목 | 내용 |
|------|------|
| **달성 조건** | **pX 플랫폼 Canton 지원 준비 완료**: pX가 Canton 위에서 결제를 생성, 전송, 수신, 완료 처리 가능 / 최소 필드: 자산(Canton Coin 및 Canton 지원 자산, USDC on Canton 등), 수량, 카운터파티, 결제일, 참조번호 / 단순 이전 및 DvP(Delivery vs Payment) 모두 지원 (카운터파티 설정이 허용하는 경우) / 내부 UAT 데모: Talos 통제 테스트 카운터파티를 이용한 엔드투엔드 흐름 |
| **검수 기준** | 외부 카운터파티 **1곳 이상**이 Canton 위에서 **프로덕션 결제 20건 이상** 완료 (CC 및 기타 Canton 자산 혼합 가능, 당일 또는 밸류데이트 결제) / 20건에 대한 기본 운영 통계 공개: 전송→확인 소요 시간, 성공/실패 건수 |
| **기한** | CIP 승인 후 **180일** |
| **획득 Weight** | **+1** |

#### D6. pX 결제 가속 보너스 (최대 +1)

| 항목 | 내용 |
|------|------|
| **달성 조건** | D5를 빨리 완료할수록 보너스 Weight를 더 받음 |
| **기한** | CIP 승인 후 **180일** |
| **획득 Weight** | 60일 이내 완료 시 **+1** (만점), 180일에 완료 시 **+0**, 선형 비례 (예: 120일 → +0.5) |

| 항목 | 내용 |
|------|------|
| **달성 조건** | Talos pX를 통해 Canton 위에서 결제된 **명목 금액(notional)** 기준으로 보너스 부여 (CC, USDC on Canton 등 모든 Canton 자산 포함) |
| **증빙 방식** | 결제 기록 집계 (지시 건수, 타임스탬프, 자산, 명목 금액) 및 대사(reconciliation) 방법론을 TWG에 제출하여 검증 |
| **기한** | pX Go Live 후 **180일** |
| **획득 Weight** | **$15억(1.5B) 명목 결제금당 +0.5**, 최대 **+1** |

### Weight 합산 구조

```
CC 트레이딩 트랙:
  D1. CC 지원 준비 완료             +1
  D2. CC 가속 보너스                최대 +1  (45일 이내 시 만점)
  D3. CC 거래소 활성화              +0.5
  D4. CC 채택 보너스                최대 +1  ($25M당 +0.5)
  소계:                            최대 3.5

pX 결제 트랙:
  D5. pX 결제 준비 완료             +1
  D6. pX 가속 보너스                최대 +1  (60일 이내 시 만점)
  D7. pX 채택 보너스                최대 +1  ($1.5B당 +0.5)
  소계:                            최대 3.0
──────────────────────────────────────────────────────────────────
총합:                                                   최대 6.5
```

> **특이 사항**: Talos CIP는 CC 트레이딩 트랙의 기한이 **90일**로, 다른 SV CIP(대부분 180일)보다 **절반으로 짧다**. 이는 Talos가 이미 기관 트레이딩 플랫폼을 운영 중이므로 기술 통합이 비교적 빠르다는 자신감의 표현으로 보인다.

## SV 보상 메커니즘 (SV Reward Mechanics)

### 에스크로 SV 설정

1. GSF(또는 GSF가 승인한 SV 노드 운영자)가 **`extraBeneficiary` PartyID**를 생성하여 에스크로 SV를 설정한다. 이때 SV Weight는 최대 획득 가능치(6.5)로 설정된다.
2. 신청자(Talos)는 에스크로 설정 과정을 GSF 및 SV 노드 운영자와 직접 조율해야 한다.
3. 에스크로 SV 운영 비용은 **신청자가 전액 부담**한다.
4. 에스크로 SV는 블록별 보상을 발행(mint)하지 않는다 — 모든 보상은 **미청구 보상 풀(Unclaimed Rewards)**로 적립된다.
5. 기존 SV 운영자의 **2/3**가 에스크로 SV 노드의 전체 Weight 호스팅을 허용하도록 설정을 업데이트한다.

### 마일스톤 달성 시

1. 신청자가 **토크노믹스 워킹그룹(TWG)**에 마일스톤 완료 증빙을 제출한다.
2. 신청자가 해당 마일스톤에 대해 받아야 할 **Canton Coin 수량 계산서**를 제시한다.
3. TWG가 마일스톤 충족을 인정하고 계산에 동의하면:
   - **Tokenomics-Announce 메일링리스트**에 공지가 발송된다.
   - GSF가 `extraBeneficiary`를 해당 SV가 통제하는 **활성 PartyID**로 변경한다.
   - SV 운영자의 **2/3**가 미청구 보상 중 일부를 신청자의 Validator에 배정한다.

### 마일스톤 미달성 시

1. GSF가 신청자에게 미달성을 통보한다.
2. `extraBeneficiary` SV에 배정된 **남은 Weight가 GSF 노드에서 제거**되고, GSF SV 노드의 총 Weight가 같은 만큼 감소한다 (SV 투표로 결정).
3. TWG가 미청구 보상 처리 방안을 SV들에 **권고**한다.

### CIP-0045 적용

- 신청자는 **CIP-0045 (SV 운영 요건)**의 적용을 받는다.
- Weight가 **2.5를 초과**하는 시점부터 **6개월 이내에 자체 SV 노드를 운영**해야 한다.
- 자체 SV 노드는 Weight 0으로 네트워크에 합류하며, 마일스톤 달성에 따라 Weight가 증가한다.
