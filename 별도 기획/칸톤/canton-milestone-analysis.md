## 2. 마일스톤 유형별 분석

### 2.1 통합(Integration) 마일스톤

모든 CIP에 공통으로 존재하는 **필수 마일스톤**이다.

| CIP | SV | 통합 내용 | Weight | 기한 |

| 0060 | Zero Hash | CC + USDC 제품 스위트 통합 | +2.5 | 180일 |
| 0072 | Fireblocks | Canton 토큰 표준 월렛 지원 | +1.0 | 180일 |
| 0074 | BitGo | Canton 토큰 표준 커스터디 지원 | +1.0 | 180일 |
| 0075 | Zodia | Off-Venue 결제(+0.5) + 3자 담보(+0.5) | +1.0 | 180일 |
| 0085 | Talos | CC 플랫폼 준비(+1) + pX 결제 준비(+1) + 거래소 활성화(+0.5) | +2.5 | 90~180일 |
| 0087 | Hex Trust | CC + 토큰 표준 세이프키핑 | +1.0 | 180일 |
| 0094 | Blockdaemon | 기관 지갑 배포 (고객 $10M+ 보유) | +1.0 | 180일 |

**패턴 요약:**
- 통합 Weight는 **+1.0이 표준** (전체의 20%)
- 여러 제품 트랙을 가진 경우 **+2.5** (Zero Hash, Talos)
- 기한은 대부분 **180일**, Talos CC 트랙만 **90일** (기존 플랫폼 활용)
- 검증: 기술 통합 완료 + 최소 1곳 고객의 MainNet 거래

### 2.2 가속 보너스(Acceleration Bonus) 마일스톤

**패턴 요약:**
- **60일/180일 선형 비례**가 표준 공식 (4/5건)
- Talos CC만 예외적으로 **45일/90일** (반감된 기한)
- 가속 보너스 **Max Weight +1~2** — 전체의 15~40%
- 보유하지 않는 SV: **Hex Trust, Zero Hash, Blockdaemon**
- 가속 보너스 = "빠른 통합을 보상" → 이미 기술적 준비가 된 SV에 유리

## 4. 검증 기준 분석

### 4.1 정량적 vs 정성적

| 유형 | 마일스톤 | 예시 |
|------|---------|------|
| **정성적** | 통합 완료 | "Canton 토큰 표준 지원 구현" |
| **정량적 + 정성적 혼합** | 통합 + 고객 실사용 | "통합 + 고객 1곳 MainNet 거래 완료" |
| **순수 정량적** | 채택 보너스 | "$1.5B 자산 이전", "10M건 거래" |

**트렌드**: 최근 CIP일수록 **정량적 검증 기준**이 강화되고 있다.

### 4.2 검증 증빙 방식

| 방식 | 사용 SV | 설명 |

## 5. 마일스톤 아키타입(Archetype) 분류

11개 SV의 마일스톤 구조를 5개 아키타입으로 분류한다.

### 아키타입 A: 통합 + 가속 + 채택 (표준 3단계)

```
[통합 +1] → [가속 보너스 max +2] → [채택 보너스 max +2] = Max 5
```

- **해당 SV**: Fireblocks (CIP-0072), BitGo (CIP-0074)
- **특징**: Eric Saraniecki가 작성한 표준 템플릿. 가장 깔끔한 구조
- **적합 대상**: 커스터디/월렛 단일 제품을 Canton에 통합하는 SV

### 아키타입 B: 2트랙 × (통합 + 가속 + 채택)

```
트랙 1: [통합 +0.5] → [가속 max +1] → [채택 max +1] = 2.5
트랙 2: [통합 +0.5] → [가속 max +1] → [채택 max +1] = 2.5
합계 = Max 5
```

- **해당 SV**: Zodia (CIP-0075), Talos (CIP-0085, 변형)
- **특징**: 2개 제품 라인을 각각 마일스톤으로 분리. Talos는 CC(3.5) + pX(3.0)으로 비대칭
- **적합 대상**: 복수 제품을 Canton에 통합하는 SV

### 아키타입 C: 통합 + 채택 (가속 없음)

```
[통합 +1] → [채택 보너스 max +2~4] = Max 3~5
```

- **해당 SV**: Hex Trust (CIP-0087), Blockdaemon (CIP-0094)
- **특징**: 가속 보너스 없이 순수 성과 연동. 단순하고 명확
- **적합 대상**: 조기 달성 인센티브가 불필요하거나 Weight가 낮은 SV

### 아키타입 D: 통합 + 교육 + 독점성 + 채택 (다면 기여)

```
[통합 +2.5] + [교육 +0.5] + [독점성 +0.5] + [채택 max +4] = Max 7.5
```

- **해당 SV**: Zero Hash (CIP-0060)
- **특징**: 기술 통합 외에 마케팅, 독점 커밋먼트를 별도 마일스톤으로 분리
- **적합 대상**: 대형 플랫폼이 Canton을 우선적으로 채택하는 전략적 파트너십

### 아키타입 E: 고정 Weight (초기 SV)

```
[고정 +1~10] = Max 1~10
```

- **해당 SV**: Copper (0015), Copper ClearLoop (0039), Deribit (0040), Circle (0041)
- **특징**: 마일스톤 구조가 없거나 PDF에만 포함. 초기 네트워크 구성 시기의 방식
- **현재 상태**: **더 이상 사용되지 않음** — 2025년 상반기부터 모든 신규 SV는 성과 연동 구조 필수

## 8. 시간에 따른 진화 트렌드

### 2024년 상반기 (초기)
- **고정 Weight**: Copper(1), Blockdaemon(1 → 이후 5로 재제안), Deribit(1)
- 마일스톤 구조 미비, PDF 중심

### 2024년 하반기
- **일괄 승인**: CIP-0032~0040 동시 투표
- 아직 간소한 구조

### 2025년 상반기 (전환기)
- **성과 연동 시작**: Circle(10, 2025-02), Zero Hash(7.5, 2025-05)
- Zero Hash에서 교육/독점성 등 **비기술 마일스톤** 등장

### 2025년 하반기 (성숙기)
- **표준 템플릿 확립**: Fireblocks(5, 08월), BitGo(5, 08월) → 동일 구조
- **복합 구조**: Zodia(5, 09월) 2트랙, Talos(6.5, 10월) 2트랙 + 7마일스톤
- **가속 보너스** 일반화 (60일/180일 선형)
- **Inelastic Burn** 등 새로운 지표 도입 (Hex Trust, 11월)

### 진화 방향 요약

```
고정 Weight → 성과 연동 → 다면 기여 + 정량 지표 강화
(2024)       (2025 상반기)  (2025 하반기)

단일 마일스톤 → 3단계 표준 → 멀티트랙 + 복합 지표
(2024)         (2025 Q3)    (2025 Q4)
```

### 9.2 권장 채택 패턴

| 웨이브릿지 상황 | 권장 아키타입 | 근거 |
|---------------|-------------|------|

| 복수 제품 (커스터디 + 결제 + RWA) | **B** (멀티트랙) | Zodia/Talos 패턴, 웨이브릿지 가치 제안에 적합 |
| 한국 시장 독점 포지셔닝 | **D** (다면 기여) | Zero Hash 패턴, 교육 + 독점성 추가 |

### 9.3 차별화 가능 요소

기존 CIP에 없는 웨이브릿지만의 차별점:
